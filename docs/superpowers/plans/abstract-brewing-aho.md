# Plan: Automated GTFS Schedule Sync Pipeline

## Context

Schedule data for CTA and Metra is currently generated by **manual npm scripts** that download GTFS zips, parse CSVs, and write results to static JSON files in `public/data/`. This means schedules go stale until someone remembers to re-run the scripts. Both transit agencies update their GTFS feeds periodically — Metra provides a `published.txt` timestamp file for change detection, and CTA's zip supports `Last-Modified`/`ETag` HTTP headers.

**Goal:** Automate this into an hourly check that detects feed updates, downloads new data, parses it, and stores it in Firestore — replacing the static JSON files with a live, self-updating data layer.

---

## Change Detection Available

| Agency    | Detection Method                                                                                                                                                                     | Feed Size |
| --------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | --------- |
| **Metra** | `https://schedules.metrarail.com/gtfs/published.txt` returns a timestamp string (e.g. `03/06/26 02:20:03 AM America/Chicago`). Also has `Last-Modified` + `ETag` headers on the zip. | ~392KB    |
| **CTA**   | No `published.txt`. The zip at `https://www.transitchicago.com/downloads/sch_data/google_transit.zip` supports `Last-Modified` + `ETag` headers via HEAD request.                    | ~99MB     |

---

## Architecture: Firebase Cloud Functions (2nd gen) + Cloud Scheduler

**Why Cloud Functions over alternatives:**

- Already using Firebase — deploys with `firebase deploy --only functions`
- Gen2 functions support 60-min timeout and up to 32GB memory (plenty for a 99MB zip)
- Cloud Scheduler trigger is configured declaratively in the function definition
- Scales to zero — essentially free for hourly HEAD requests + rare full syncs
- Admin SDK has automatic Firestore access (no extra credentials needed)
- `METRA_API_TOKEN` is already in Cloud Secret Manager

**Two separate scheduled functions:**

| Function        | Schedule                    | Memory | Purpose                                                      |
| --------------- | --------------------------- | ------ | ------------------------------------------------------------ |
| `syncCtaGtfs`   | `0 * * * *` (top of hour)   | 1GB    | Check CTA Last-Modified header → download + parse if changed |
| `syncMetraGtfs` | `5 * * * *` (:05 past hour) | 512MB  | Check Metra published.txt → download + parse if changed      |

Separated to avoid memory overlap (CTA zip is 250x larger) and for independent failure handling.

## Firestore Collection Structure (replaces static JSON)

```
gtfs-meta/                              # Change detection state
  cta                                   # { lastModified, etag, lastCheckedAt, lastSyncedAt }
  metra                                 # { publishedTimestamp, lastModified, etag, lastCheckedAt, lastSyncedAt }

schedules/                              # Per-station schedules (replaces public/data/*-schedules/)
  {station-slug}                        # { service, directions: [{ headsign, line, weekday, saturday, sunday }] }
                                        # ~336 docs (99 CTA + 237 Metra), largest ~12KB

metra-trips/                            # Individual trip details (replaces public/data/metra-trip-detail/)
  {trip-id}                             # { tripId, trainNumber, headsign, line, stops: [...] }
                                        # ~3,431 docs, each ~2KB

metra-trip-indexes/                     # Trip index per line (replaces public/data/metra-trip-index/)
  {line-slug}                           # { weekday: [...], saturday: [...], sunday: [...] }
                                        # 11 docs, each ~40KB

metra-station-trips/                    # Trips per station (replaces public/data/metra-station-trips/)
  {station-slug}                        # { weekday: [...], saturday: [...], sunday: [...] }
                                        # ~237 docs, largest ~193KB (under 1MB Firestore limit)
```

**Total:** ~4,015 documents, ~25MB storage — well within Firestore free tier.

## Frontend Migration: API Routes

Client components currently fetch static JSON (`/data/cta-schedules/{slug}.json`). Moving to Firestore requires server-side API routes since client components can't use `firebase-admin`.

**New API routes:**

| Route                                         | Replaces                                      | Consumer               |
| --------------------------------------------- | --------------------------------------------- | ---------------------- |
| `app/api/schedules/[slug]/route.ts`           | `public/data/{service}-schedules/{slug}.json` | `Arrivals.tsx`         |
| `app/api/metra/station-trips/[slug]/route.ts` | `public/data/metra-station-trips/{slug}.json` | `StationTimetable.tsx` |
| `app/api/metra/trip-index/[line]/route.ts`    | `public/data/metra-trip-index/{slug}.json`    | Trip index pages       |
| `app/api/metra/trips/[tripId]/route.ts`       | `public/data/metra-trip-detail/{id}.json`     | Trip detail pages      |

Each route: read one Firestore doc, return JSON with `Cache-Control: public, s-maxage=3600, stale-while-revalidate=86400`.

The server component `metra/[line]/train/[tripId]/page.tsx` (currently uses `fs.readFileSync`) migrates to a direct Firestore admin SDK read — cleaner than the file approach.

## Implementation Phases

### Phase 1: Shared Library Extraction

Extract duplicated parsing logic from the 3 generate scripts into a shared `functions/src/lib/` library:

- `gtfs-utils.ts` — `downloadBuffer()`, `parseGTFS()`, `timeToMinutes()`
- `parsers/cta-schedules.ts` — CTA GTFS → schedule data structures
- `parsers/metra-schedules.ts` — Metra GTFS → schedule data structures
- `parsers/metra-trips.ts` — Metra GTFS → trip details, indexes, station trips
- `writers/firestore-writer.ts` — batched Firestore writes (500-op chunks)

**Critical source files:**

- `scripts/generate-cta-schedules.ts`
- `scripts/generate-metra-schedules.ts`
- `scripts/generate-metra-trips.ts`
- `scripts/seed-stations.ts`

### Phase 2: Cloud Functions + Change Detection

- Create `functions/` directory with `package.json`, `tsconfig.json`
- Implement `change-detection.ts` (HEAD requests + published.txt)
- Implement `syncCtaGtfs` and `syncMetraGtfs` scheduled functions
- Add `gtfs-meta` collection for tracking sync state
- Update `firebase.json` to include functions config
- Deploy with `firebase deploy --only functions`

### Phase 3: API Routes + Frontend Migration

- Create the 4 new API routes listed above
- Update `Arrivals.tsx` to fetch from `/api/schedules/{slug}`
- Update `StationTimetable.tsx` to fetch from `/api/metra/station-trips/{slug}`
- Update `metra/[line]/train/[tripId]/page.tsx` to read from Firestore directly
- Update any other components fetching from `public/data/`

### Phase 4: Cleanup

- Remove `public/data/cta-schedules/`, `public/data/metra-schedules/`, `public/data/metra-trip-detail/`, `public/data/metra-trip-index/`, `public/data/metra-station-trips/`
- Update `CLAUDE.md` and `DATA_SOURCES.md` to document new architecture
- Optionally keep old generate scripts as manual fallback or remove them

### Phase 5: Station Sync (stretch goal)

- After parsing GTFS stops, compare against existing Firestore stations
- Auto-create new stations, log removed ones (don't auto-delete — accumulated metadata)
- Update `lineOrder` for affected stations
- Write sync results to a `sync-logs` collection for observability

## Verification Plan

1. **Unit test** the shared parsing library against a snapshot of real GTFS data
2. **Local emulator test** — run functions against Firebase emulator to verify Firestore writes
3. **Deploy to staging** — trigger functions manually via `firebase functions:shell` and verify:
   - `gtfs-meta` docs update correctly
   - `schedules` collection has correct data for a known station
   - `metra-trips` collection has correct trip details
4. **Frontend verification** — confirm `Arrivals.tsx` and `StationTimetable.tsx` render correctly from API routes
5. **End-to-end** — wait for the next real GTFS update, verify the function detects it and syncs automatically
