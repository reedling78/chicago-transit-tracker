# Articles Section — Design Spec

**Date:** 2026-04-03
**Status:** Approved

---

## Overview

Add an editorial articles/blog section to Chicago Transit Tracker so transit guides, explainers, and stories can be published alongside the existing CTA and Metra data pages. Articles are authored as MDX files in the repo, processed at build time, and served as static HTML — consistent with the site's existing `output: 'export'` architecture.

A companion Claude Code skill (`article-writer`) guides authors through the full workflow: research, metadata, writing, image processing, and file creation.

---

## Problem / Motivation

The site currently surfaces structured transit data (lines, stations, schedules) but has no room for narrative content — guides like "How to use the CTA from O'Hare" or "What is Metra and who is it for?" These are high-value SEO pages that also serve real users. An articles section fills that gap without requiring a CMS or any runtime infrastructure.

---

## Goals

- Authors can write and publish articles using only Claude Code and a text editor
- Articles are statically rendered — no CMS, no database, no runtime server
- Draft and scheduled publishing are supported at build time
- One article can be designated as "featured" on the list page
- The system is simple enough to onboard a new team member in a single session

## Non-Goals

- No comments, likes, or social features
- No tag or category archive pages (list page is flat for now)
- No search within articles
- No CMS UI — MDX files in the repo are the source of truth

---

## Architecture

### Content storage

Articles live as MDX files in the repo:

```
content/
  articles/
    {slug}/
      {slug}.mdx        ← frontmatter + MDX body
public/
  images/
    articles/
      {slug}/
        {slug}.{ext}    ← original source image
        {slug}-og.{ext} ← 1200×630, center-cropped
        {slug}-twitter.{ext} ← 1200×675, center-cropped
```

This keeps content and code together, enables PR-based review of articles, and eliminates any external dependency.

### Build-time rendering

`app/lib/articles.ts` reads MDX files from `content/articles/` at build time using Node `fs`. Frontmatter is parsed with `gray-matter`. MDX body is compiled in React Server Components using `next-mdx-remote/rsc`. No runtime server is involved — fully compatible with `output: 'export'`.

### Filtering

Two visibility controls are evaluated at build time:

- `draft: true` — always excluded from the build output
- `publishAt` — if set and in the future relative to build time, excluded

This means scheduling a post requires a rebuild at or after the publish time. Given the CI/CD pipeline deploys on every push to `main`, this is a non-issue in practice.

### Featured post

`featuredPost: true` in frontmatter marks the article as the featured card on the list page. Only one article should carry this flag at a time. The `article-writer` skill handles unsetting it on other articles when a new featured post is written.

## Routes

| Route              | Page               | Notes                              |
| ------------------ | ------------------ | ---------------------------------- |
| `/articles`        | Article list       | Featured card + chronological list |
| `/articles/[slug]` | Individual article | Full MDX render with metadata      |

Both routes follow the existing SEO pattern: `generateMetadata` with `title`, `description`, `openGraph`, and `twitter` fields. The individual article uses `type: 'article'` in openGraph for better social sharing previews.

## Key Design Decisions

**`next-mdx-remote` over `@next/mdx`**
`@next/mdx` requires MDX files to live inside `app/` as route files. Placing content in `content/articles/` (separate from application code) is cleaner and matches the convention used by most editorial Next.js sites. `next-mdx-remote/rsc` reads files from anywhere on disk and compiles them in RSC — no plugin needed, no change to `next.config.ts`.

**`rehype-raw` for inline HTML in MDX**
The `article-writer` skill uses `<span data-term class="underline decoration-dotted">` to mark technical terms. Without `rehype-raw`, MDX strips raw HTML. Adding this plugin is the minimal fix.

**`@tailwindcss/typography` for prose styling**
Rather than hand-rolling article body typography, the `prose` utility class handles headings, paragraphs, lists, blockquotes, and code blocks consistently. Loaded via Tailwind v4's `@plugin` directive — no `tailwind.config.js` required.

**`imageVersion` as a cache-busting timestamp**
Hero images are served as static files. When an image is updated, the URL must change to bust CDN and browser caches. The `imageVersion` frontmatter field (a Unix millisecond timestamp auto-generated by the skill at write time) is appended as a query string: `?v={imageVersion}`.

## Skill: `article-writer`

A project-local Claude Code skill at `.claude/skills/article-writer/SKILL.md` guides the end-to-end article workflow in six steps:

1. **Collect inputs** — subject, angle, source URLs (each fetched for research context)
2. **Collect metadata** — title, description, slug, category, tags, author, date, publishAt, featuredPost
3. **Propose plan** — angle/thesis, section structure, tone, word count target (700–1000)
4. **Write article** — prose-first, no emoji, authoritative tone; `<span data-term>` for technical terms
5. **Process images** — ImageMagick commands to produce original, OG, and Twitter crops
6. **Write to disk** — assemble full MDX with frontmatter, handle featuredPost conflicts, write file

The skill enforces no-AI-phrase voice rules and the prose-first structure without the author needing to remember them.

## Component Strategy

Reuse existing components wherever possible:

- `PageHeader` — article title and description on the detail page
- `Breadcrumb` — `Home → Articles → {title}` on the detail page
- `LinkCard` — article list items (title, description excerpt, date + category as meta)

New for this feature:

- Featured article card on the list page (larger visual treatment with hero image)
- `prose dark:prose-invert` wrapper div on the article detail page for MDX body styling

---

## Sitemap

Per the standing rule, both `/articles` and each `/articles/{slug}` route are added to `app/sitemap.ts`. The list page uses `changeFrequency: 'weekly'`; individual articles use `'monthly'`.
